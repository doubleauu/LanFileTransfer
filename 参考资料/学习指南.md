# 局域网文件传输系统学习指南

本文面向刚学完 C# 网络应用编程课程的初学者，帮助你按顺序读懂并复现本项目。建议先把项目跑起来，再按功能一块一块看代码；不要一开始就从第一行看到最后一行，否则很容易被 WinForms 控件、TCP、EF Core 和文件流同时绕晕。

本指南参考了项目根目录的 `教学ppt总结.md`，重点对应其中这些章节：

| 课程章节 | 本项目中的对应内容 |
|---|---|
| 1. C#、.NET 与 Visual Studio 基础 | 解决方案、项目引用、类、枚举、属性、命名空间 |
| 2. 控制台与 WinForms 入门 | 服务端 Console、客户端 WinForms 界面 |
| 3. 文件读写与数据流 | `FileStream` 分块上传和下载文件 |
| 4. 数据库、DataGridView、LINQ 与 EF Core | SQLite 数据库、实体类、查询文件列表、表格显示 |
| 5. IP、DNS、进程与线程 | IP、端口、服务端后台任务、WinForms 异步操作 |
| 6. TCP 应用编程 | `TcpListener`、`TcpClient`、`NetworkStream`、长度前缀协议 |
| 8. 综合项目思维 | 从界面操作到协议命令，再到服务端处理和数据库保存 |

## 1. 先建立整体印象

这个项目叫“局域网文件传输系统”，核心目标是：客户端选择文件，发送给服务端；服务端保存文件，并把文件信息写进数据库；其他客户端可以从服务端查看列表并下载。

可以把它理解成四层：

```text
WinForms 客户端
    ↓ TcpClient + NetworkStream
Console 服务端
    ↓ EF Core
SQLite 数据库
    ↓
服务端本地文件目录
```

每一层做的事情很明确：

| 层 | 主要职责 | 先看文件 |
|---|---|---|
| 客户端界面 | 收集用户输入、显示按钮、表格、进度和日志 | `LanFileTransfer.Client/Form1.cs` |
| 客户端通信 | 连接服务端、发命令、传文件内容 | `LanFileTransfer.Client/ClientConnection.cs` |
| 公共协议 | 定义命令类型、命令拼接、命令读取 | `LanFileTransfer.Common/CommonModels.cs`、`TcpMessageProtocol.cs` |
| 服务端处理 | 接收连接、识别命令、处理业务 | `LanFileTransfer.Server/Program.cs`、`ClientHandler.cs` |
| 数据库 | 保存用户、文件信息、传输记录 | `LanFileTransfer.Data/LanFileTransferDbContext.cs` |

初学时不要急着记住所有代码。先记住一句话：

```text
用户点按钮 → 客户端发命令 → 服务端处理命令 → 数据库/文件系统保存结果 → 服务端回响应 → 客户端更新界面
```

## 2. 推荐学习顺序

### 第 1 步：会运行项目

先做到“能看见程序工作”，再去理解代码。

1. 用 Visual Studio 打开 `LanFileTransfer.sln`。
2. 先启动 `LanFileTransfer.Server`。
3. 再启动 `LanFileTransfer.Client`。
4. 客户端服务器 IP 填 `127.0.0.1`，端口填 `5000`。
5. 点击“连接测试”。
6. 注册一个用户。
7. 登录。
8. 上传一个小文本文件。
9. 刷新文件列表。
10. 下载刚才上传的文件。
11. 查看传输记录。

这一步的学习目标不是看懂全部代码，而是知道系统完整流程长什么样。

### 第 2 步：看项目结构

从解决方案层面理解为什么要分四个项目：

```text
LanFileTransfer.Client      客户端，负责界面和用户操作
LanFileTransfer.Server      服务端，负责监听、处理请求、保存文件
LanFileTransfer.Common      公共内容，客户端和服务端都要用
LanFileTransfer.Data        数据库上下文和实体类
```

这里最重要的是引用关系：

```text
Client → Common
Server → Common
Server → Data
Data   → Common
```

客户端不直接访问数据库。原因是客户端只负责发请求，服务端才是真正管理数据和文件的一方。这样更像真实的 C/S 架构。

### 第 3 步：先学公共协议

建议先读：

```text
LanFileTransfer.Common/CommonModels.cs
LanFileTransfer.Common/TcpMessageProtocol.cs
```

`CommonModels.cs` 里主要是枚举：

| 枚举 | 作用 |
|---|---|
| `MessageType` | 区分登录、注册、上传、下载、文件列表等命令 |
| `ResourceType` | 区分普通文件和文件夹 ZIP |
| `TransferType` | 区分上传记录和下载记录 |
| `TransferStatus` | 保存传输是否成功 |

`TcpMessageProtocol.cs` 是本项目最关键的基础代码。它解决的是 TCP 的一个核心问题：**TCP 是字节流**，没有天然的“**消息边界**”。

如果直接发送：

```text
LoginRequest|zhangsan|123456
```

服务端不能保证一次 `Read` 就刚好读到这一整条命令。可能一次只读到一半，也可能把两条命令读到一起。因此项目使用：

```text
4 字节命令长度 + UTF-8 命令字符串
```

学习时重点看三个方法：

| 方法 | 学习重点 |
|---|---|
| `SendCommandAsync` | 先把命令转成字节，再写入长度和内容 |
| `ReceiveCommandAsync` | 先读 4 字节长度，再按长度读完整命令 |
| `ReadExactAsync` | 用循环读取解决一次读不完整的问题 |

这一部分对应 `教学ppt总结.md` 的“6.3 解决 TCP 无消息边界”。

### 第 4 步：理解服务端启动

读：

```text
LanFileTransfer.Server/Program.cs
LanFileTransfer.Server/ServerDatabase.cs
```

`Program.cs` 做三件事：

```text
初始化数据库
创建 TcpListener
循环等待客户端连接
```

核心流程可以理解为：

```csharp
TcpListener listener = new(IPAddress.Any, port);
listener.Start();

while (true)
{
    TcpClient client = await listener.AcceptTcpClientAsync();
    _ = Task.Run(() => ClientHandler.HandleClientAsync(client));
}
```

`IPAddress.Any` 表示服务端监听本机所有网卡地址。局域网里其他电脑访问服务端时，用的是服务端电脑的局域网 IP。

`Task.Run` 表示每来一个客户端连接，就交给一个后台任务处理。这个项目没有继续做复杂的多客户端管理，但已经能说明基础并发思想。

### 第 5 步：理解客户端界面和按钮事件

读：

```text
LanFileTransfer.Client/Program.cs
LanFileTransfer.Client/Form1.cs
```

`Program.cs` 只负责启动 WinForms 主窗体。真正的界面和按钮逻辑都在 `Form1.cs`。

你可以按按钮顺序看：

| 按钮 | 事件方法 | 对应功能 |
|---|---|---|
| 连接测试 | `BtnConnect_Click` | 发 `TestRequest` |
| 注册 | `BtnRegister_Click` | 发 `RegisterRequest` |
| 登录 | `BtnLogin_Click` | 发 `LoginRequest` |
| 上传文件 | `BtnUploadFile_Click` | 选择文件并上传 |
| 上传文件夹 | `BtnUploadFolder_Click` | 压缩为 ZIP 后上传 |
| 刷新列表 | `BtnRefreshFiles_Click` | 发 `FileListRequest` |
| 下载文件 | `BtnDownloadFile_Click` | 发 `DownloadRequest` |
| 传输记录 | `BtnTransferRecords_Click` | 发 `TransferRecordRequest` |

WinForms 学习重点：

- 控件对象：`TextBox`、`Button`、`Label`、`ProgressBar`、`DataGridView`。
- 事件处理：按钮点击后进入对应的 `Click` 方法。
- 异步操作：用 `async` 和 `await` 避免界面卡死。
- 表格显示：用 `DataGridView.Rows.Add(...)` 把服务端返回的数据放到界面中。

### 第 6 步：理解客户端通信类

读：

```text
LanFileTransfer.Client/ClientConnection.cs
```

这个类是为了让窗体代码不要直接处理太多网络细节。

主要方法：

| 方法 | 作用 |
|---|---|
| `SendCommandAsync` | 建立连接，发送普通命令，接收响应 |
| `UploadFileAsync` | 发送上传命令，再发送文件字节 |
| `DownloadFileAsync` | 发送下载命令，先收响应，再收文件字节 |
| `SendFileContentAsync` | 用 `FileStream` 分块读取本地文件并写入网络流 |
| `ReceiveFileContentAsync` | 从网络流分块读取文件并写到本地 |

文件上传不是把文件内容塞进命令字符串，而是：

```text
UploadRequest|用户Id|资源类型|文件大小|文件名|扩展名
后面紧跟文件字节内容
```

这样做的好处是文件再大也可以分块传输，不需要一次性读入内存。

### 第 7 步：理解服务端业务处理

读：

```text
LanFileTransfer.Server/ClientHandler.cs
```

这个文件是服务端的业务核心。学习时先看最上面的 `HandleClientAsync`：

```text
接收命令
判断命令类型
调用不同处理方法
```

对应关系如下：

| 命令 | 服务端方法 |
|---|---|
| `TestRequest` | 直接返回测试响应 |
| `RegisterRequest` | `HandleRegisterAsync` |
| `LoginRequest` | `HandleLoginAsync` |
| `UploadRequest` | `HandleUploadAsync` |
| `FileListRequest` | `HandleFileListAsync` |
| `DownloadRequest` | `HandleDownloadAsync` |
| `TransferRecordRequest` | `HandleTransferRecordsAsync` |

学习诀窍：不要一次看完整个 `ClientHandler.cs`。按功能看，例如今天只看注册登录，明天只看上传，后天只看下载。

## 3. 核心功能逐个学

### 3.1 连接测试

连接测试是最小 TCP 流程。

```text
客户端点击连接测试
↓
客户端发送 TestRequest
↓
服务端接收并返回 TestResponse
↓
客户端在日志框显示响应
```

重点理解：

- 客户端用 `TcpClient.ConnectAsync` 连接服务端。
- 服务端用 `TcpListener.AcceptTcpClientAsync` 接收连接。
- 双方通过 `NetworkStream` 读写数据。

建议练习：

- 把端口改错，观察客户端提示。
- 关闭服务端后点击连接测试，观察异常提示。
- 在服务端控制台看是否输出客户端连接日志。

### 3.2 注册和登录

注册流程：

```text
客户端发送 RegisterRequest|用户名|密码
↓
服务端判断用户名是否重复
↓
服务端保存新用户
↓
服务端返回 RegisterResponse
```

登录流程：

```text
客户端发送 LoginRequest|用户名|密码
↓
服务端查询用户
↓
服务端比较密码哈希
↓
登录成功后返回用户 Id 和用户名
```

重点理解：

- 数据库里不直接保存明文密码，而是保存 `SHA256` 哈希。
- `FirstOrDefaultAsync` 用来查询用户。
- `AnyAsync` 用来判断用户名是否已存在。
- 登录成功后，客户端保存 `currentUserId`，后续上传、下载、查列表都要用它。

建议练习：

- 注册同名用户，观察服务端返回。
- 登录错误密码，观察提示。
- 登录成功后再上传文件，理解为什么上传需要用户 Id。

### 3.3 单文件上传

上传是本项目最重要的功能之一。

流程：

```text
客户端选择文件
↓
客户端读取文件名、扩展名、文件大小
↓
客户端发送 UploadRequest
↓
客户端分块发送文件内容
↓
服务端分块接收并写入临时文件
↓
服务端校验收到的字节数
↓
服务端移动到正式目录
↓
服务端写入 FileRecord 和 TransferRecord
↓
服务端返回 UploadResponse
```

重点理解：

- 文件元数据走命令字符串。
- 文件内容走 `NetworkStream` 的字节流。
- `FileStream.ReadAsync` 每次读一块。
- `NetworkStream.WriteAsync` 每次发一块。
- 服务端先写临时文件，成功后再移动到正式目录。
- 服务端保存文件名用 `Guid`，避免重名覆盖。

建议练习：

- 上传一个很小的 `.txt` 文件。
- 上传一个中文文件名文件。
- 上传两个同名文件，观察服务端是否覆盖。
- 找到服务端运行目录里的 `ServerStorage`，看看文件实际保存在哪里。

### 3.4 文件列表查询

文件列表是数据库查询和表格显示的结合。

流程：

```text
客户端发送 FileListRequest|用户Id
↓
服务端查询 FileRecords
↓
服务端把多条文件记录拼成响应字段
↓
客户端解析字段
↓
客户端填充 DataGridView
```

重点理解：

- 文件内容不放数据库，数据库只保存文件元数据。
- `Include(file => file.Uploader)` 用来同时取上传者信息。
- `OrderByDescending` 让新上传的文件排在前面。
- 客户端通过循环读取响应字段，每 6 个字段是一条文件记录。

建议练习：

- 上传多个文件后刷新列表。
- 看列表中的文件 ID、文件名、大小、类型、上传者和上传时间。
- 对照 `ShowFileListResponse` 理解表格是怎么填出来的。

### 3.5 单文件下载

下载和上传方向相反。

流程：

```text
客户端选择文件
↓
客户端发送 DownloadRequest|用户Id|文件Id
↓
服务端查询数据库并找到文件路径
↓
服务端先返回 DownloadResponse
↓
服务端分块发送文件内容
↓
客户端分块接收并保存到本地
↓
服务端保存下载记录
```

重点理解：

- 客户端必须先收到下载响应，知道文件大小后才能按指定字节数接收。
- 下载时也不能一次性读完整文件，要循环读写。
- 客户端保存文件使用 `SaveFileDialog`。
- 下载完成后，服务端会保存一条 `TransferRecord`。

建议练习：

- 下载一个文本文件并打开对比内容。
- 下载中文文件名文件。
- 下载后查看传输记录是否新增。

### 3.6 文件夹上传和下载

TCP 传输的是字节流，不适合直接“传一个文件夹”。所以项目采用课堂上更容易理解的办法：

```text
文件夹 → ZIP 压缩包 → 按普通文件上传
```

上传文件夹流程：

```text
选择文件夹
↓
客户端用 ZipFile.CreateFromDirectory 生成临时 ZIP
↓
上传 ZIP
↓
服务端记录 ResourceType = FolderZip
↓
客户端删除临时 ZIP
```

下载文件夹流程：

```text
下载 ZIP
↓
如果资源类型是 FolderZip
↓
询问是否解压
↓
用 ZipFile.ExtractToDirectory 解压
```

重点理解：

- 文件夹传输本质上仍然是文件传输。
- `ResourceType` 只是告诉系统这个文件代表一个文件夹压缩包。
- ZIP 解压在客户端完成，服务端只负责保存和发送 ZIP 文件。

建议练习：

- 上传一个包含子文件夹的目录。
- 下载后选择解压。
- 检查解压后的目录结构是否和原来一致。

### 3.7 传输记录

传输记录是“业务功能 + 数据库查询 + 表格显示”的综合练习。

上传和下载成功后，服务端都会保存一条 `TransferRecord`：

| 字段 | 含义 |
|---|---|
| `UserId` | 哪个用户操作 |
| `FileId` | 操作的是哪个文件 |
| `TransferType` | 上传或下载 |
| `Status` | 是否成功 |
| `BytesTransferred` | 实际传输字节数 |
| `ClientIp` | 客户端地址 |
| `StartedAt` / `FinishedAt` | 开始和结束时间 |

客户端点击“传输记录”后，服务端只查询当前用户的记录，然后返回给客户端表格显示。

## 4. 数据库部分怎么学

先读：

```text
LanFileTransfer.Data/LanFileTransferDbContext.cs
```

本项目数据库里有三张主要表：

```text
User
FileRecord
TransferRecord
```

关系可以理解成：

```text
一个用户可以上传多个文件
一个用户可以有多条传输记录
一个文件可以有多条传输记录
```

EF Core 学习重点：

| 代码 | 含义 |
|---|---|
| `DbContext` | 数据库上下文，代表一次数据库操作入口 |
| `DbSet<User>` | 对应数据库中的用户表 |
| `HasIndex(...).IsUnique()` | 给用户名加唯一约束 |
| `HasConversion<string>()` | 把枚举保存成字符串 |
| `HasOne(...).WithMany(...)` | 配置表之间的一对多关系 |
| `SaveChangesAsync()` | 把内存中的修改保存到数据库 |

本项目为了课程展示简单，启动时用 `EnsureCreated()` 创建数据库表。正式项目一般会用 Migrations，但大作业阶段先理解实体、表、查询和保存更重要。

## 5. TCP 和文件流的重点概念

### 5.1 TCP 不是“一条消息一条消息”传

初学者最容易误会：

```text
我发送一次，服务端就一定接收一次。
```

实际上 TCP 是连续字节流。一次 `Read` 可能读到半条命令，也可能读到多条命令的一部分。所以本项目用长度前缀：

```text
先发 4 字节长度
再发命令内容
接收方先读长度
再循环读够指定长度
```

记住：`ReadExactAsync` 是理解 TCP 拆包问题的关键。

### 5.2 命令字符串只传“说明书”

例如上传文件时，命令字符串只说明：

```text
谁上传
上传的是文件还是文件夹 ZIP
文件有多大
原文件名是什么
扩展名是什么
```

真实文件内容随后用字节流传输。

### 5.3 大文件必须分块

如果一次性把大文件读进内存，文件很大时程序可能卡住或占用太多内存。

所以项目使用：

```text
64KB 缓冲区
循环 ReadAsync
循环 WriteAsync
累计已传输字节数
更新进度条
```

这对应 `教学ppt总结.md` 的“3.2 FileStream 分块读取”和“3.3 NetworkStream 读写”。

## 6. 建议的七天学习安排

如果你想一点点学，可以按这个节奏：

| 天数 | 学习内容 | 目标 |
|---|---|---|
| 第 1 天 | 运行项目、看 README、看界面 | 知道系统能做什么 |
| 第 2 天 | 看 `CommonModels.cs` 和 `TcpMessageProtocol.cs` | 理解命令类型、长度前缀、字段转义 |
| 第 3 天 | 看服务端 `Program.cs`、客户端 `ClientConnection.cs` | 理解 TCP 连接和命令收发 |
| 第 4 天 | 看注册登录相关代码 | 理解用户表、密码哈希、登录状态 |
| 第 5 天 | 看上传相关代码 | 理解文件分块、临时文件、数据库记录 |
| 第 6 天 | 看文件列表和下载相关代码 | 理解数据库查询、表格显示、下载保存 |
| 第 7 天 | 看文件夹 ZIP 和传输记录 | 完整串起项目，准备答辩讲解 |

每学完一天，建议写 3 句话：

```text
今天看懂了哪个功能？
这个功能从客户端到服务端经过了哪些方法？
还有哪一行代码不理解？
```

这样比单纯“看过代码”更有效。

## 7. 动手练习清单

### 基础练习

- 修改服务端端口为 `6000`，客户端也改成 `6000` 后重新连接。
- 注册两个不同用户，分别上传文件。
- 上传中文文件名文件，确认列表显示正常。
- 上传同名文件两次，确认服务端没有覆盖。
- 下载文件后对比原文件内容。

### 代码阅读练习

- 在 `TcpMessageProtocol.SendCommandAsync` 打断点，看发送的命令文本。

## 8. 常见问题和排查方法

### 连接不上服务端

按顺序检查：

1. 服务端是否先启动。
2. 客户端 IP 是否正确。
3. 端口是否一致，默认是 `5000`。
4. 如果跨电脑测试，Windows 防火墙是否拦截。
5. 服务端控制台是否输出“服务端已启动”。

### 上传失败

按顺序检查：

1. 是否已经登录。
2. 文件是否正在被其他程序占用。
3. 服务端是否有写入目录权限。
4. 服务端控制台是否输出异常。
5. `ServerStorage/temp` 是否残留临时文件。

### 下载失败

按顺序检查：

1. 是否先刷新了文件列表。
2. 是否选中了文件列表而不是传输记录表格。
3. 服务端数据库里的文件路径是否仍然存在。
4. 服务端 `ServerStorage` 中实际文件是否被删除。

### 中文显示异常

本项目 Markdown 文档是 UTF-8 编码。读取中文文件时要使用 UTF-8；如果看到乱码，不要猜内容，换编码方式确认。

## 9. 答辩时可以这样讲

### 项目总体介绍

本项目是一个基于 C/S 架构的局域网文件传输系统。客户端使用 WinForms 提供图形界面，服务端使用 Console 程序监听 TCP 连接。客户端和服务端通过 `TcpClient`、`TcpListener`、`NetworkStream` 通信，服务端使用 EF Core 和 SQLite 保存用户、文件信息和传输记录，真实文件保存在服务端本地目录。

### 技术亮点

可以重点讲这几项：

1. 使用 4 字节长度前缀解决 TCP 粘包和拆包问题。
2. 文件内容使用 `FileStream` 和 `NetworkStream` 分块传输，避免一次性读入大文件。
3. 文件先写入临时目录，校验大小成功后再移动到正式目录。
4. 服务端用 GUID 生成保存文件名，避免重复文件名覆盖。
5. 使用 EF Core + SQLite 保存用户、文件元数据和传输记录。
6. 文件夹先压缩为 ZIP，再复用普通文件上传下载流程。
7. 客户端使用 `async` / `await`，上传下载时界面不会长时间卡死。

### 演示顺序

建议按这个顺序演示：

```text
启动服务端
启动客户端
连接测试
注册用户
登录
上传普通文件
刷新文件列表
下载普通文件
上传文件夹
下载并解压文件夹 ZIP
查看传输记录
展示服务端数据库和保存目录
```

### 容易被问到的问题

| 问题 | 回答思路 |
|---|---|
| 为什么不用 UDP？ | 文件传输要求可靠、有序，TCP 更适合。 |
| 为什么命令前面要加 4 字节长度？ | TCP 没有消息边界，长度前缀能让接收方知道一条命令读到哪里结束。 |
| 为什么数据库不直接保存文件内容？ | 文件内容大，放磁盘更合适；数据库只保存文件名、路径、大小等元数据。 |
| 为什么要分块传输？ | 大文件一次性读入内存会占用太多资源，分块更稳定。 |
| 为什么文件夹要压缩成 ZIP？ | 网络传输本质上传的是字节，ZIP 可以把文件夹结构打包成一个普通文件。 |
| 为什么服务端保存文件名用 GUID？ | 避免不同用户上传同名文件时互相覆盖。 |
